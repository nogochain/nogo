### 1.2 设计原则

- **模块化**: 各模块职责单一，松耦合

- **可观测**: 完整的日志和监控体系

## 2. 核心模块

### 2.1 区块链核心模块 (`blockchain/`)

**职责**: 实现区块链的核心功能，包括共识、交易处理、区块验证等。

#### 2.1.1 共识引擎

**文件**: `consensus.go`, `miner.go`, `validator.go`

**功能**:
- NogoPow 工作量证明算法
- 难度动态调整
- 区块验证和奖励计算
- 叔块奖励机制

**关键参数**:
```go
TargetBlockTimeSec = 17          // 目标出块时间（秒）
MaxBlockTimeDriftSec = 7200      // 最大时间漂移（2 小时）
DifficultyWindow = 10            // 难度调整窗口
```

#### 2.1.2 P2P 网络

**文件**: `p2p_server.go`, `p2p_client.go`, `p2p_manager.go`, `peers.go`

**功能**:
- 节点发现和连接管理
- 消息编码和解码
- 节点评分系统
- DDoS 防护

**配置**:
```go
DefaultP2PMaxConnections = 100   // 最大连接数
MaxPeersDiscoverPerRound = 10    // 每轮发现节点数
```

#### 2.1.3 交易处理

**文件**: `mempool.go`, `txid.go`, `types.go`

**功能**:
- 交易验证
- 内存池管理
- 交易排序
- 批量签名验证

**限制**:
```go
MaxTxPerBlockDefault = 100       // 每区块最大交易数
MempoolMaxSize = 10000           // 内存池最大交易数
```

#### 2.1.4 存储系统

**文件**: `store.go`, `store_bolt.go`

**功能**:
- BoltDB 数据库操作
- 区块和交易存储
- 余额和状态管理
- 检查点机制

**存储模式**:
- **Pruned**: 保留最近 1000 个区块
- **Full**: 保留所有区块

### 2.2 加密库 (`internal/crypto/`)

**职责**: 提供加密原语和安全功能。

**算法**:
- **Ed25519**: 数字签名
- **SHA256**: 哈希函数
- **Keccak256**: POW 计算

**优化**:
- 批量签名验证（阈值：10 个签名）
- 对象池复用
- Merkle 树验证

### 2.3 网络通信 (`internal/networking/`)

**职责**: 实现安全的网络通信协议。

**组件**:
- `secure_channel.go`: 安全通道建立
- `identity.go`: 节点身份管理
- `codec.go`: 消息编码
- `ratelimit.go`: 速率限制

### 2.4 存储和缓存 (`internal/storage/`, `internal/cache/`)

**职责**: 提供高效的数据存储和缓存机制。

**存储**:
- BoltDB 嵌入式数据库
- 数据迁移支持
- 存储模式配置

**缓存**:
- LRU 缓存实现
- Block 缓存：10,000 个区块
- Balance 缓存：100,000 个账户
- Proof 缓存：10,000 个证明

### 2.5 API 层 (`api/`)

**职责**: 提供 HTTP 和 WebSocket 接口。

**HTTP API**:
- RESTful 端点
- CORS 支持
- 速率限制
- 管理员令牌认证

**WebSocket**:
- 实时事件订阅
- 地址订阅
- 心跳保活

### 2.6 配置和日志 (`config/`, `internal/logger/`, `internal/metrics/`)

**配置管理**:
- 环境变量注入
- JSON 配置文件
- 命令行标志

**日志系统**:
- 结构化日志
- 多级别支持（Debug/Info/Warn/Error）
- JSON 格式输出

**监控指标**:
- Prometheus metrics
- 链高和交易数
- P2P 连接状态
- HTTP 请求延迟
- POW 缓存命中率

## 5. 数据流

### 5.1 交易提交流程

```
1. 用户签名交易
   ↓
2. HTTP API 接收交易
   ↓
3. 验证交易签名和格式
   ↓
4. 添加到内存池
   ↓
5. 矿工打包交易到区块
   ↓
6. 区块验证和广播
   ↓
7. 更新状态和余额
   ↓
8. 持久化到数据库
```

### 5.2 区块同步流程

```
1. 从 P2P 网络接收新区块
   ↓
2. 验证区块头和 POW
   ↓
3. 验证区块中的交易
   ↓
4. 检查是否形成最长链
   ↓
5. 如果是最长链：应用区块并更新状态
   ↓
6. 否则：存储为旁支区块
   ↓
7. 广播给其他节点
```

### 5.3 共识流程

```
1. 矿工监听内存池
   ↓
2. 收集交易构建候选区块
   ↓
3. 计算 POW（NogoPow 算法）
   ↓
4. 找到有效 nonce
   ↓
5. 广播新区块
   ↓
6. 其他节点验证
   ↓
7. 验证通过后添加到链
```

## 7. 最佳实践

### 7.1 安全配置

1. **启用 TLS**: 生产环境强制使用 HTTPS
2. **强管理员令牌**: 使用随机生成的令牌
3. **速率限制**: 配置合理的速率限制
4. **防火墙规则**: 仅开放必要端口

### 7.2 性能优化

1. **SSD 存储**: 使用 NVMe SSD 提升 I/O 性能
2. **内存调优**: 根据可用内存调整缓存大小
3. **并发配置**: 调整 worker 数量和批处理大小
4. **网络优化**: 使用高质量网络线路

### 7.3 监控告警

1. **关键指标**:
   - 链高停滞检测
   - 错误率监控
   - P2P 连接数
   - 内存使用率

2. **告警规则**:
   - 链高超过 5 分钟未增长
   - 错误率超过 1%
   - 内存使用率超过 80%
