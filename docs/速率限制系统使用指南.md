# 增强速率限制系统使用指南

## 概述

增强的速率限制系统基于令牌桶算法实现，支持每端点独立配置、IP 级别和用户级别的速率限制，以及 API 密钥认证后的限制提升。

## 主要特性

### 1. 令牌桶算法 (Token Bucket)
- **令牌生成**: 按固定速率生成令牌（每秒 RequestsPerSecond 个）
- **突发支持**: 允许突发流量（Burst 容量）
- **自动补充**: 令牌随时间自动补充
- **并发安全**: 使用互斥锁保证线程安全

### 2. 每端点独立配置
不同 API 端点可以配置不同的速率限制：
```json
{
  "rate_limits": {
    "default": {
      "requests_per_second": 10,
      "burst": 20
    },
    "/api/tx": {
      "requests_per_second": 50,
      "burst": 100
    },
    "/api/balance": {
      "requests_per_second": 20,
      "burst": 40
    }
  }
}
```

### 3. API 密钥认证提升
- **默认提升倍数**: 5 倍（可配置）
- **支持过期时间**: API 密钥可设置过期时间
- **支持吊销**: 可随时吊销 API 密钥
- **分层支持**: 支持不同等级（basic, premium, enterprise）

### 4. 速率限制头信息
所有响应包含以下头信息：
- `X-RateLimit-Limit`: 最大请求数（桶容量）
- `X-RateLimit-Remaining`: 剩余请求数（当前令牌数）
- `X-RateLimit-Reset`: 重置时间（Unix 时间戳）
- `Retry-After`: 重试等待时间（秒，仅在限流时返回）

### 5. Prometheus 监控指标
- `nogo_rate_limit_requests_total`: 总请求数（按端点和结果分类）
- `nogo_rate_limit_tokens_remaining`: 剩余令牌数
- `nogo_rate_limit_current_rps`: 当前 RPS 限制
- `nogo_rate_limit_bypasses_total`: API 密钥绕过次数
- `nogo_rate_limit_events`: 限流事件
- `nogo_active_rate_limiters`: 活跃限流器数量

## 配置示例

### 基本配置
```go
cfg := &RateLimitConfig{
    Default: &EndpointRateLimitConfig{
        RequestsPerSecond: 10.0,  // 每秒 10 个请求
        Burst:             20,     // 突发容量 20
    },
    Endpoints: map[string]*EndpointRateLimitConfig{
        "/api/tx": {
            RequestsPerSecond: 50.0,
            Burst:             100,
        },
    },
    APIKeyMultiplier: 5.0,        // API 密钥提升 5 倍
    Enabled:          true,       // 启用限流
    ByIP:             true,       // 按 IP 限流
    ByUser:           false,      // 不按用户限流
    StorageType:      "memory",   // 内存存储
    CleanupInterval:  time.Minute, // 清理间隔
}
```

### 环境变量配置
```bash
# 启用速率限制
export API_RATE_LIMIT_ENABLED=true

# 默认 RPS 和突发
export API_RATE_LIMIT_RPS=10
export API_RATE_LIMIT_BURST=20

# API 密钥提升倍数
export API_RATE_LIMIT_API_KEY_MULTIPLIER=5.0

# 信任代理（获取真实 IP）
export API_TRUST_PROXY=true
```

## 使用示例

### 1. 创建速率限制器
```go
import "github.com/nogochain/nogo/blockchain/api"

// 创建配置
cfg := api.DefaultRateLimitConfig()
cfg.Enabled = true
cfg.StorageType = "memory"

// 创建限流器
rateLimiter, err := api.NewEnhancedRateLimiter(cfg)
if err != nil {
    log.Fatalf("Failed to create rate limiter: %v", err)
}
defer rateLimiter.Stop()
```

### 2. 注册 API 密钥
```go
// 注册 API 密钥
apiKey := "your-api-key-here"
err := rateLimiter.RegisterAPIKey(
    apiKey,           // API 密钥
    "user123",        // 所有者
    "premium",        // 等级
    5.0,              // 提升倍数
    time.Time{},      // 过期时间（零表示永不过期）
)
if err != nil {
    log.Fatalf("Failed to register API key: %v", err)
}
```

### 3. 集成到 HTTP 中间件
```go
// 创建 HTTP 中间件
middleware := api.RateLimitMiddleware(
    rateLimiter,
    // 获取客户端 IP
    func(r *http.Request) string {
        return api.ExtractClientIP(r, true) // 信任代理
    },
    // 获取 API 密钥
    func(r *http.Request) string {
        return api.ExtractAPIKey(r, "X-API-Key", "api_key")
    },
    // 获取用户 ID（可选）
    func(r *http.Request) string {
        return r.Header.Get("X-User-ID")
    },
)

// 应用到 HTTP 服务器
handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
    fmt.Fprint(w, "Success")
})

httpServer := &http.Server{
    Addr:    ":8080",
    Handler: middleware(handler),
}
```

### 4. 直接使用限流器
```go
// 检查请求是否允许
endpoint := "/api/tx"
identifier := "192.168.1.1" // IP 或用户 ID
apiKey := "optional-api-key"

allowed, retryAfter, limit, remaining := rateLimiter.Allow(endpoint, identifier, apiKey)

if !allowed {
    // 返回 429 Too Many Requests
    w.Header().Set("Retry-After", strconv.Itoa(int(retryAfter.Seconds())))
    w.WriteHeader(http.StatusTooManyRequests)
    return
}

// 设置限流头
w.Header().Set("X-RateLimit-Limit", strconv.FormatFloat(limit, 'f', 0, 64))
w.Header().Set("X-RateLimit-Remaining", strconv.FormatFloat(remaining, 'f', 0, 64))
```

### 5. 等待直到允许
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

err := rateLimiter.WaitUntilAllowed(ctx, "/api/tx", "192.168.1.1", "")
if err != nil {
    // 超时或其他错误
    log.Printf("Wait failed: %v", err)
    return
}

// 现在可以发送请求
```

### 6. 批量获取令牌
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

// 尝试一次性获取 10 个令牌
err := rateLimiter.AcquireTokens(ctx, "/api/batch", "192.168.1.1", 10)
if err != nil {
    log.Printf("Failed to acquire tokens: %v", err)
    return
}

// 执行批量操作
```

### 7. 管理 API 密钥
```go
// 获取 API 密钥信息
info, err := rateLimiter.GetAPIKeyInfo(apiKey)
if err != nil {
    log.Printf("Invalid API key: %v", err)
}

// 吊销 API 密钥
err = rateLimiter.RevokeAPIKey(apiKey)
if err != nil {
    log.Printf("Failed to revoke API key: %v", err)
}

// 生成新的 API 密钥
newKey, err := api.GenerateAPIKey()
if err != nil {
    log.Fatalf("Failed to generate API key: %v", err)
}
```

### 8. 获取统计信息
```go
stats := rateLimiter.GetStats()
fmt.Printf("Enabled: %v\n", stats["enabled"])
fmt.Printf("Active Endpoints: %v\n", stats["active_endpoints"])
fmt.Printf("API Keys Count: %v\n", stats["api_keys_count"])
fmt.Printf("Default RPS: %v\n", stats["default_rps"])
fmt.Printf("Default Burst: %v\n", stats["default_burst"])
```


### 查询示例
```promql

# 限流拒绝率
sum(rate(nogo_rate_limit_requests_total{result="denied"}[5m]))

# API 密钥使用率

# 各端点令牌剩余情况
nogo_rate_limit_tokens_remaining
```

## 最佳实践

### 1. 配置建议
- **默认端点**: 设置较低的默认限制（如 10 RPS）
- **高频端点**: 为查询类端点设置较高限制（如 50-100 RPS）
- **写操作端点**: 为写入类端点设置较低限制（如 5-10 RPS）
- **突发容量**: 设置为 RPS 的 2-5 倍

### 2. API 密钥管理
- **定期轮换**: 建议每 3-6 个月轮换一次
- **最小权限**: 根据用户需求分配适当的提升倍数
- **监控使用**: 通过 Prometheus 指标监控 API 密钥使用情况

### 3. 监控告警
```yaml
# Prometheus 告警规则示例
groups:
- name: rate_limiting
  rules:
  - alert: HighRateLimitRejection
    expr: sum(rate(nogo_rate_limit_requests_total{result="denied"}[5m])) > 100
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High rate limit rejection rate"
      
  - alert: RateLimiterMemoryHigh
    expr: nogo_active_rate_limiters > 10000
    for: 10m
    labels:
      severity: info
    annotations:
      summary: "Too many active rate limiters"
```

### 4. 性能优化
- **内存存储**: 默认使用内存存储，性能最优（< 1ms 开销）
- **Redis 存储**: 分布式场景可选 Redis（待实现）
- **清理间隔**: 根据内存使用情况调整清理间隔
- **并发控制**: 限流器本身已做并发优化

## 注意事项

1. **并发安全**: 限流器内部已处理并发，可安全地在多个 goroutine 中使用
2. **内存管理**: 定期清理不活跃的限流桶（默认 10 分钟 TTL）
3. **时间精度**: 令牌补充基于时间戳计算，精度为纳秒
4. **溢出保护**: 所有数学运算都有溢出检查
5. **错误处理**: 所有错误都包含上下文信息，支持 errors.Is/As

## 故障排除

### 问题：限流器不工作
- 检查 `cfg.Enabled` 是否设置为 `true`
- 确认中间件已正确应用到 HTTP 服务器
- 检查配置是否通过验证（`cfg.Validate()`）

### 问题：API 密钥未生效
- 确认 API 密钥已正确注册
- 检查 API 密钥是否过期或被吊销
- 验证请求头名称是否正确（默认 `X-API-Key`）

### 问题：内存使用过高
- 减少 `CleanupInterval` 间隔
- 降低 `DefaultTTL` 时间
- 监控 `nogo_active_rate_limiters` 指标
