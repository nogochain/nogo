## 1. Overview

### 1.1 What is NogoChain

NogoChain is a decentralized Layer 1 public blockchain utilizing the original NogoPow (Nogo Proof of Work) consensus algorithm.

**Core Features**:
- **NogoPow Consensus**: Matrix computation-based PoW algorithm with ASIC-resistant properties
- **Ed25519 Signatures**: Ed25519 digital signatures for transaction security
- **P2P Network**: Decentralized mesh network topology
- **Smart Monetary Policy**: Geometric decay block reward mechanism
- **High Concurrency**: Support for batch transaction verification and parallel processing

### 1.2 Technology Stack

- **Programming Language**: Go 1.24.0
- **Database**: BoltDB (embedded KV store)
- **Cryptography**: Ed25519, SHA256, SHA3-256
- **Network Protocol**: Custom JSON-based P2P protocol
- **Monitoring**: Prometheus + Grafana

### 1.3 Project Structure

```
nogo/
├── blockchain/          # Core blockchain implementation
│   ├── core/           # Core data structures
│   ├── consensus/      # Consensus validator
│   ├── nogopow/        # NogoPow engine
```

## 2. Quick Start

### 2.1 Requirements

- Go 1.24.0 or higher
- OS: Linux / macOS / Windows
- RAM: Minimum 2GB, Recommended 4GB+
- Storage: SSD recommended

### 2.2 Installation

```bash
# Clone repository
git clone https://github.com/nogochain/nogo.git
cd nogo

# Install dependencies
go mod download

# Build
go build -o nogo ./cmd/node
```

```bash

./nogo server NOGO<your_address> mine

# Testnet node
./nogo server NOGO<your_address> mine test
```

### 3.2 Module Responsibilities

#### Core Layer
- **Block**: Block data structures and operations
- **Transaction**: Transaction data structures and validation
- **Account**: Account model and state management

#### Consensus Layer
- **Validator**: Block validator, executes all validation rules
- **NogoPow**: PoW engine, implements mining and validation
- **Fork Detector**: Fork detection and resolution

#### Network Layer
- **P2P Server**: P2P server, handles node connections
- **Sync**: Block synchronization, downloads and validates historical blocks
- **Peer Manager**: Peer management, maintains peer list and scoring

#### Storage Layer
- **BoltDB Store**: BoltDB-based persistent storage
- **Cache**: LRU cache for hot data acceleration
- **Checkpoint**: Checkpoint mechanism for fast sync

### 3.3 Data Flow

#### Block Production Flow
```
1. Miner selects transactions from mempool
   ↓
2. Creates candidate block (including coinbase)
   ↓
3. NogoPow engine computes PoW (finds valid nonce)
   ↓
4. After finding valid nonce, broadcasts block
   ↓
5. Other nodes validate block
   ↓
6. After validation, adds to local chain
   ↓
7. Updates state, confirms transactions
```

#### Transaction Processing Flow
```
1. User signs transaction
   ↓
2. Submits to node API (HTTP/WS)
   ↓
3. Node validates transaction (signature, nonce, balance)
   ↓
4. Adds to mempool
   ↓
5. Broadcasts to other nodes
   ↓
6. Miner includes in block
   ↓
7. After block confirmation, transaction completes
```

```go
address := core.GenerateAddress(pubKey)
```

**Validate Address**:

```go
err := core.ValidateAddress("NOGO...")
```

### 4.2 Account

Each account contains:
- **Balance**: Account balance (uint64, smallest unit: wei)
- **Nonce**: Transaction counter (prevents replay attacks)

```go
type Account struct {
```

### 4.3 Transaction

Transaction types:
- **TxCoinbase**: Block reward transaction (no inputs, only outputs)
- **TxTransfer**: Regular transfer transaction

```go
type Transaction struct {
    Type      TransactionType `json:"type"`
```

**Transaction Fields**:
- **Type**: Transaction type (coinbase/transfer)

- **Amount**: Transfer amount (wei)
- **Fee**: Transaction fee (wei)
- **Nonce**: Sender's transaction counter
- **Data**: Additional data (optional)
- **Signature**: Ed25519 signature (64 bytes)

## 5. Block Structure

### 5.1 Block Composition

Each block contains:

1. **Block Header**: 112 bytes
   - Version (4 bytes): Block version
   - PrevHash (32 bytes): Parent block hash
   - TimestampUnix (8 bytes): Unix timestamp
   - DifficultyBits (4 bytes): Difficulty target
   - Difficulty (4 bytes): Difficulty value
   - Nonce (8 bytes): PoW nonce
   - MerkleRoot (32 bytes): Transaction Merkle root

2. **Transaction List**: Variable length
   - Coinbase transaction (must be first)
   - Regular transfer transactions

3. **Metadata**:
   - Hash: Block hash
   - Height: Block height
   - MinerAddress: Miner address
   - TotalWork: Cumulative work

### 5.2 Block Hash Calculation

Block Hash = SHA256(Block Header Serialized)

```go
func (b *Block) CalculateHash() []byte {
    headerBytes := b.Header.Serialize()
    hash := sha256.Sum256(headerBytes)
    return hash[:]
}
```

### 5.3 Merkle Tree

Merkle tree is used for efficient transaction inclusion verification:

```
        Root (MerkleRoot)
       /    \
      /      \
    Hash01   Hash23
   /    \    /    \
 Hash0  Hash1 Hash2 Hash3
  (Tx0) (Tx1) (Tx2) (Tx3)
```

**Calculation**:
```go
leaves := [][]byte{tx0.Hash(), tx1.Hash(), tx2.Hash(), tx3.Hash()}
merkleRoot, _ := core.MerkleRoot(leaves)
```

## 6. Transaction Mechanism

### 6.1 Transaction Creation

```go
// Create transfer transaction
tx := core.Transaction{
    Type:      core.TxTransfer,
    ChainID:   1,
    FromPubKey: pubKey,
    ToAddress: "NOGO...",
    Amount:    1000000000000000000, // 1 NOGO
    Fee:       1000000,             // 0.001 NOGO
    Nonce:     1,
    Data:      "",
}

// Calculate signing hash
signHash, _ := tx.SigningHash()

// Sign
signature := ed25519.Sign(privateKey, signHash)
tx.Signature = signature

// Verify transaction
err := tx.Verify()
```

### 6.2 Transaction Validation

Validation flow:

1. **Basic Validation**:
   - Valid transaction type
   - Amount > 0
   - Valid address format

2. **Signature Validation**:
   - Public key length = 32 bytes
   - Signature length = 64 bytes
   - Ed25519 verification passes

3. **Nonce Validation**:
   - Nonce > 0
   - Nonce = Account's current nonce + 1

4. **Balance Validation**:
   - Account balance >= Amount + Fee

### 6.3 Transaction Fees

Fee market mechanism:
- **Minimum Fee**: Configured by node (default 1 wei)
- **Fee Priority**: Mempool sorted by fee
- **RBF (Replace-By-Fee)**: Supports fee replacement

```go
// Mempool sorted by fee descending
entries := mempool.EntriesSortedByFeeDesc()

// RBF replacement
newTx.Fee = oldTx.Fee + 1000000 // Increase fee
txid, replaced, _, _ := mempool.ReplaceByFee(newTx)
```

## 7. Consensus Algorithm

### 7.1 NogoPow Algorithm

NogoPow is NogoChain's original PoW algorithm, combining matrix operations and hash functions.

**Algorithm Flow**:

```
Input: Block header hash blockHash, seed (parent block hash)
Output: PoW hash powHash

1. Get matrix data from cache
   cacheData = Cache.Get(seed)

2. Construct input matrix
   A = blockHash converted to matrix

3. Matrix multiplication
   Result = A × cacheData

4. Hash result
   powHash = SHA3-256(Result)

5. Difficulty check
   Requirement: powHash < target (difficulty target)
```

**Mining Code**:
```go
engine := nogopow.New(nogopow.DefaultConfig())

header := &nogopow.Header{
    Number:     big.NewInt(height),
    Time:       uint64(timestamp),
    ParentHash: parentHash,
    Difficulty: difficulty,
    Coinbase:   minerAddress,
}

// Mining loop
for nonce := uint64(0); ; nonce++ {
    header.Nonce = nonce
    
    // Check if valid solution found
    if engine.VerifySeal(header) == nil {
        // Found valid block
        break
    }
}
```

### 7.2 Difficulty Adjustment

Difficulty adjustment goal: Maintain stable block time at target interval (default 10 seconds)

**Adjustment Formula**:
```
newDifficulty = parentDifficulty × (targetTime / actualTime)
```

**Adjustment Limits**:
- Maximum increase: 200%
- Maximum decrease: 50%
- Minimum difficulty: Configured MinimumDifficulty

```go
adjuster := nogopow.NewDifficultyAdjuster(config)
newDiff := adjuster.CalcDifficulty(currentTime, parentHeader)
```

### 7.3 Fork Selection

**Longest Chain Rule** (actually maximum cumulative work chain):

```go
func SelectBestChain(chains []*Chain) *Chain {
    var best *Chain
    var bestWork *big.Int
    
    for _, chain := range chains {
        work := chain.TotalWork()
        if best == nil || work.Cmp(bestWork) > 0 {
            best = chain
            bestWork = work
        }
    }
    
    return best
}
```

**Fork Resolution**:
- Automatically switch to chain with greater cumulative work
- Limit reorganization depth (default 100 blocks)
- Trigger reorg when necessary, rollback state

## 8. Network Protocol

### 8.1 P2P Message Types

| Message Type | Direction | Description |
|-------------|-----------|-------------|
| `hello` | Bidirectional | Handshake message |
| `chain_info_req` | Client→Server | Request chain info |
| `chain_info` | Server→Client | Return chain info |
| `headers_from_req` | Client→Server | Request block headers |
| `headers` | Server→Client | Return block header list |
| `block_by_hash_req` | Client→Server | Request block by hash |
| `block` | Server→Client | Return block |
| `tx_req` | Client→Server | Request transaction |
| `tx_broadcast` | Broadcast | Broadcast transaction |
| `block_broadcast` | Broadcast | Broadcast block |
| `getaddr` | Client→Server | Get peer list |
| `addr` | Server→Client | Return peer list |

### 8.2 Handshake Protocol

```json
// Client sends
{
  "type": "hello",
  "payload": {
    "protocol": 1,
    "chainId": 1,
    "rulesHash": "abc123...",
    "nodeId": "NOGO..."
  }
}

// Server responds
{
  "type": "hello",
  "payload": {
    "protocol": 1,
    "chainId": 1,
    "rulesHash": "abc123...",
    "nodeId": "NOGO..."
  }
}
```

**Handshake Validation**:
1. Protocol version matches (protocol = 1)
2. Chain ID consistent (chainId matches)
3. Rules hash consistent (rulesHash matches)

### 8.3 Block Synchronization

**Sync Flow**:

```
1. Get chain info
   GET chain_info_req
   → Returns current height, latest hash

2. Request block headers
   GET headers_from_req (from=0, count=100)
   → Returns 100 block headers

3. Validate block headers
   - Verify PoW
   - Verify difficulty
   - Verify timestamp

4. Request block bodies
   GET block_by_hash_req (hash)
   → Returns full block

5. Validate and store block
   - Verify transactions
   - Apply state transitions
   - Persist storage

6. Repeat steps 2-5 until sync complete
```

### 8.4 Peer Management

**Peer Scoring Dimensions**:
- **Success Rate**: Successful requests / Total requests
- **Response Time**: Average response latency
- **Activity**: Last seen time
- **Uptime**: Cumulative online time

**Peer Selection Strategy**:
- Prioritize high-score peers
- Limit connections per IP
- Regularly clean up low-quality peers

## 9. Economic Model

### 9.1 Monetary Policy

**Block Reward Formula**:
```
R(Y) = R₀ × (1 - r/10)^Y

Where:
- R₀: Initial block reward
- r:  Annual reduction rate (AnnualReductionPercent)
- Y:  Years elapsed
- R(Y): Block reward in year Y
```

**Example Parameters**:
```go
policy := MonetaryPolicy{
    InitialBlockReward:     1000 * 1e18,  // 1000 NOGO
    MinimumBlockReward:     10 * 1e18,    // 10 NOGO
    AnnualReductionPercent: 10,           // 10% annual reduction
    MinerFeeShare:          100,          // Miner gets 100% of fees
}
```

**Reward Calculation**:
```go
reward := policy.BlockReward(height)
// Year 1: 1000 NOGO
// Year 2: 900 NOGO
// Year 3: 810 NOGO
// ...
// Minimum: 10 NOGO
```

### 9.2 Fee Distribution

**Miner Revenue = Block Reward + Transaction Fees**

```go
// Calculate miner fee revenue
feeAmount := policy.MinerFeeAmount(totalFees)

// Calculate total reward (including uncle reward)
totalReward := policy.GetTotalMinerReward(height, uncleCount)
```

### 9.3 Uncle Mechanism

**Uncle Reward**:
- 1 block from main chain: 7/8 block reward
- 2 blocks from main chain: 6/8 block reward
- ...
- 7 blocks from main chain: 1/8 block reward
- Distance ≥ 8 blocks: No reward

**Inclusion Reward**:
- Block including uncle gets bonus: 1/32 block reward/uncle
- Maximum 2 uncles per block

### 9.4 Inflation Control

**Supply Formula**:
```
S(Y) = S(Y-1) + R(Y) × N

Where:
- S(Y): Total supply at end of year Y
- N:   Blocks per year (approximately 3,153,600)
```

**Inflation Rate Trend**:
- Year 1: High inflation (network launch incentive)
- Years 2-5: Decreasing inflation rate
- Year 10+: Approaches zero (minimum reward)

## 12. FAQ

### 12.1 General Questions

**Q: What is NogoChain's block time?**
A: Target block time is 10 seconds, achieved through dynamic difficulty adjustment.

**Q: What is the total supply?**
A: No hard cap, but inflation rate decreases over time, eventually approaching the minimum block reward.

**Q: How to become a validator node?**
A: Run a full node and enable mining to participate in consensus.

### 12.2 Technical Questions

**Q: Node sync is slow, what to do?**
A: 
1. Use SSD storage
2. Increase P2P connections
3. Enable checkpoint sync

**Q: Transaction not confirming, what to do?**
A: 
1. Check if transaction fee is sufficient
2. Use RBF to increase fee
3. Contact miners for priority inclusion

**Q: How to backup node data?**
A: Backup the `DATA_DIR` directory, which contains the complete blockchain data.

### 12.3 Development Questions

**Q: Which programming languages are supported?**
A: Official SDKs for Go, Python, JavaScript; community contributes other languages.

**Q: How to test smart contracts?**
A: Deploy and test on testnet, get test coins from faucet.

**Q: Are there API rate limits?**
A: Public nodes have rate limits; running your own node is recommended.
